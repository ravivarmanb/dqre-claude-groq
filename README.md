# DQ Engine · UK Health Insurance
### LangGraph + ChromaDB + Gemini Flash 2.5 + SQLite

---

## Architecture

```
Knowledge Source (PDF/HTML/TXT/Text)
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│                    LangGraph Pipeline                       │
│                                                             │
│  [load_feedback] ──▶ [extract_knowledge] ──▶ [generate]    │
│       │                                         │           │
│  Reads past feedback                      Gemini Flash 2.5  │
│  from SQLite                              generates rules   │
│                                                             │
│  [deduplicate] ◀──────────────────────────────────┘        │
│       │                                                     │
│  ChromaDB cosine                                            │
│  similarity check                                           │
│       │                                                     │
│  [sync_rules] ──▶ SQLite dq_rules table + ChromaDB         │
└─────────────────────────────────────────────────────────────┘

Approval Workflow:
  PENDING_APPROVAL ──▶ ACTIVE (approve) or REJECTED (reject)
  ACTIVE ──▶ DEACTIVATED (deactivate with active_to date)
  DEACTIVATED ──▶ ACTIVE (reactivate, resets active_from)

Execution Pipeline:
┌─────────────────────────────────────────────────────────────┐
│  [load_feedback] ──▶ [execute_rules]                        │
│                            │                                │
│                    For each ACTIVE rule:                    │
│                    - Check active_from/active_to dates      │
│                    - Execute SQL template against SQLite     │
│                    - Calculate pass_rate                     │
│                    - Compare vs threshold_pct               │
│                    - Store in rule_executions               │
└─────────────────────────────────────────────────────────────┘

Feedback Loop:
  User rates execution ──▶ execution_feedback table
  Next generation run reads feedback ──▶ Gemini adjusts rules
```

## Database Schema

### Health Insurance Domain
- **members** — NHS number, DOB, postcode, status
- **policies** — type, premium, excess, start/end dates
- **claims** — treatment, provider, amounts, status
- **providers** — NHS registration, type
- **premiums** — payment tracking
- **waitlists** — treatment wait times

### DQ Framework
- **dq_rules** — rule_id, rule_seq, sql_template, active_from, active_to, status
- **rule_executions** — pass_rate, failed_records, run_id
- **execution_feedback** — feedback_type, comment, suggested_threshold
- **rule_runs** — summary of each run batch

## Rule Lifecycle

```
Generated by AI
      │
      ▼
PENDING_APPROVAL  ← New rules land here
      │
  ┌───┴───┐
  ▼       ▼
ACTIVE  REJECTED
  │
  ▼
DEACTIVATED (active_to set to today)
  │
  ▼
ACTIVE (reactivated via UI)
```

## Synthetic Data Issues (intentional for DQ testing)

The seed data includes deliberate quality issues:
- MEM005: Missing NHS number
- MEM007: Future date of birth (2050)
- POL005: Negative premium (−£10)
- CLM005: Claim linked to unregistered provider
- CLM006: Claim date before treatment date
- CLM008: Claim on lapsed policy
- POL009: Missing excess amount
- PRM006: Negative premium payment

## API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/api/generate` | Run generation pipeline |
| POST | `/api/execute` | Execute all active rules |
| GET | `/api/rules` | List all rules |
| POST | `/api/rules/:id/approve` | Approve rule |
| POST | `/api/rules/:id/reject` | Reject rule |
| POST | `/api/rules/:id/deactivate` | Deactivate (set active_to) |
| POST | `/api/rules/:id/reactivate` | Reactivate rule |
| GET | `/api/executions` | Execution results |
| GET | `/api/runs` | Run history |
| POST | `/api/feedback` | Submit execution feedback |
| GET | `/api/stats` | Dashboard statistics |
